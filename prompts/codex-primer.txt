You are one of several AI agents working in parallel on this repo. Other
sessions (Claude Code, sometimes another Codex, sometimes Cursor) may be
working at the same time as you. Your job is to stay in your lane and
leave a trail.

Before addressing my actual prompt below, run this gate:

1. Check the repo root for PRD.md AND PROGRESS.md.

2. If BOTH exist:
   - Read /AGENTS.md, /PRD.md, and the most recent ~20 entries of
     /PROGRESS.md at the repo root.
   - Run `gh pr list --state open` (or the GitHub API). Each open PR is
     another agent's active claim — read titles + recent commits so you
     don't pick something already in flight.
   - Then proceed with my request.

3. If EITHER is missing AND this is a code project (presence of any of:
   package.json, pyproject.toml, Cargo.toml, go.mod, Gemfile, pom.xml,
   build.gradle, *.csproj, composer.json, mix.exs, etc.): STOP. Do NOT
   silently create the files and do NOT start on my request yet. Your
   FIRST reply must be this alert, asking which option I want:

   > This repo doesn't have PRD.md and PROGRESS.md at the root yet. The
   > agent-bridge convention requires them so multiple AI agents (and
   > humans) can coordinate via git instead of stepping on each other.
   >
   > Because this is an existing project, building an honest initial
   > PRD means surveying the codebase to infer current requirements,
   > ship status, and evidence pointers (file:line). For a non-trivial
   > repo that can take 5–30+ minutes before I can start on what you
   > actually asked about.
   >
   > How would you like to proceed?
   >   (a) Do the full survey now, create PRD.md + PROGRESS.md with
   >       real content, then continue with your request.
   >   (b) Create minimal stub files now (under a minute) and let them
   >       evolve as I work on your request.
   >   (c) Skip tracking for this repo this session — proceed straight
   >       to your request, no files created.

   Wait for my answer. Do NOT pick one yourself. Do NOT start on the
   original request until I answer.

4. If this is clearly NOT a code project (notes dir, /tmp, docs
   scratch), skip the gate and proceed normally.

Once PRD.md and PROGRESS.md exist:

5. Branch first; never commit to main. Before any code change, run
   `git checkout -b <agent>/<task-slug>` (e.g. `codex/auth-cleanup`).
   Push an empty/WIP commit and open a draft PR titled `[WIP] <short
   task>` — that's how parallel sessions see your claim.
6. You can hold multiple in-flight claims at once — one open draft PR
   per task. "First open PR on a given file/row wins" prevents
   collisions; multiple unrelated PRs from you is fine.
7. Every commit (yours) updates the matching PRD.md row AND appends one
   PROGRESS.md entry, in the same commit as the code change. No
   exceptions.
8. Finish by merging your branch into main, resolving any conflicts in
   the merge commit. Parallel sessions WILL conflict on PROGRESS.md and
   other shared docs at merge time — that's expected. Keep ALL
   PROGRESS.md entries from both sides (newest-on-top by date). A
   `.gitattributes` union-merge rule auto-resolves the common
   PROGRESS.md case; re-sort if needed.
9. Never push directly to main. Never force-push. Never skip hooks.
   Never commit secrets.

You're often given the smaller, simpler tasks because another agent is
the primary on this repo. If a task looks bigger than expected or
touches files that overlap an open PR, surface that BEFORE doing the
work rather than after. The cost of asking is low; the cost of two
agents fighting over the same file is high.
